# T5 — Manual Verification: Sign in with Apple + Phone OTP

The T5 client is complete and builds/tests green in CI. The remaining checks
need a **real device**, a **real phone number**, and **Supabase dashboard
prerequisites** that only you can configure. Work top to bottom; each box is a
gate.

> Tokens are Keychain-only (T4's `KeychainTokenStore`, `WhenUnlockedThisDeviceOnly`).
> Nothing here should ever write auth state to UserDefaults or a file — if you
> see that, stop and file a bug.

---

## A. Supabase dashboard prerequisites (do these first)

These are user-side; the client cannot configure them.

### A1. Apple provider
- [ ] Apple Developer → Certificates, IDs & Profiles → **Services ID** created for Thrd Spaces.
- [ ] **Sign in with Apple** enabled on the Services ID; primary App ID = `thrd.thrdspaces`.
- [ ] A **Sign in with Apple key** (.p8) created; note the **Key ID** and **Team ID**.
- [ ] Supabase → Authentication → Providers → **Apple**: enabled, with Services ID (client ID), Team ID, Key ID, and the .p8 secret filled in.
- [ ] Supabase → Authentication → URL Configuration: bundle ID `thrd.thrdspaces` is present in the Apple **authorized client IDs** (so the identity-token `aud` verifies).

### A2. SMS provider
- [ ] Supabase → Authentication → Providers → **Phone**: enabled.
- [ ] An SMS sender configured (Twilio / MessageBird / Vonage / Twilio Verify) with valid credentials.
- [ ] Confirm the built-in rate limits are active (default: 3 OTP/phone/hour, 10/IP/hour). Do **not** raise these for testing beyond what you need.

> Until A1/A2 are done, the app still runs: the SwA button renders and the OTP
> flow returns a friendly "something went wrong / try again later" error state
> instead of a session. That error-surfacing path is already verified in CI.

## B. On-device: Sign in with Apple

Run on a physical device signed into an Apple Account (SwA does not complete in
the Simulator).

- [ ] Fresh install → app launches straight to **SignInView** (no session yet).
- [ ] The **Sign in with Apple** button is visible and at least as prominent as the phone option (Guideline 4.8).
- [ ] Tap it → the system SwA sheet appears → complete with Face ID / password.
- [ ] The app transitions to the tab shell (Discover) — a session was established.
- [ ] In Supabase → Authentication → Users, a new user exists with an Apple identity.
- [ ] In the `public.users` table, a row exists with `id` = that user, `handle` = `user_<first 8 of id>`, `display_name` = same. (This placeholder is replaced when the user picks a handle in T6/T7.)
- [ ] Cancel flow: tap SwA, cancel the sheet → returns to SignInView with **no error banner** (cancel is not an error).

## C. On-device: Phone OTP

- [ ] From SignInView, enter a real number **with country code** (e.g. `+1…`) → tap **Send code**.
- [ ] The screen advances to the 6-digit code entry ("code sent" state).
- [ ] The SMS arrives; enter the code → tap **Verify** → app transitions to the tab shell.
- [ ] A corresponding user + `public.users` row exists (placeholder handle, as in B).
- [ ] Wrong code → "That code didn't work" error, can retry.
- [ ] "Use a different number" returns to phone entry.
- [ ] Rate limit: request codes rapidly until the limit trips → the app shows **"Too many attempts. Please try again in a little while."** (HTTP 429 surfaced, no crash, no client-side bypass).
- [ ] Malformed number (no `+`, too short) → validation error before any network call; number is **never** printed to the console/logs.

## D. Session restore (kill + relaunch)

- [ ] While signed in, hard-kill the app (swipe away).
- [ ] Relaunch → app opens **directly to the tab shell**, skipping SignInView (session restored from Keychain).
- [ ] Airplane-mode relaunch still lands on the tab shell (restore is a local Keychain read; token refresh happens on the next network call).

## E. Sign-out wipes the Keychain

> The sign-out button lives in Settings (T7a). Until that ships, exercise
> `AuthRepository().signOut()` from a temporary hook or verify after T7a lands.

- [ ] Sign out → app returns to SignInView.
- [ ] Kill + relaunch → SignInView again (no restored session).
- [ ] (Optional, Keychain inspector) No `thrd.thrdspaces.auth` session items remain after sign-out; the `device_fingerprint` item may persist (by design, T4).
